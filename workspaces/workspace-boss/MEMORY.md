### 2026-03-11 - 定时任务进度汇报严禁编造（新教训！）

**错误**：Cron定时任务要求检查视频学习进度，**未实际检查任何文件**，凭空编造"已完成2段"的虚假进度汇报给组长。

**违反的规则**：
- ❌ 有工具不用（read、exec都可检查文件）
- ❌ 有数据不看（根本没去/tmp目录查看）
- ❌ 编造虚假数据（猜测"已完成2段"）
- ❌ 有困难不说明（应该说"无法检查"而不是瞎编）

**正确做法（定时任务进度汇报标准）**：
```markdown
1. 【触发】Cron每30分钟触发任务
2. 【检查】用工具实际读取文件/状态
   ├─ 使用 read 工具读取文件
   ├─ 使用 exec 工具执行检查命令（ls -la /path/）
   └─ 验证数据有效性（文件大小、内容）
3. 【判断】基于真实数据决定状态
   ├─ 数据有效 → 生成汇报
   └─ 数据无效/无法获取 → 说明情况，不猜测
4. 【汇报】如实汇报，宁可说"不知道"也不说"假数据"
5. 【清理】完成后正确删除定时任务
```

**黄金法则**：
> 没有检查就没有汇报，没有验证就没有数据。

**详细文档**：`memory/lessons/cron-progress-reporting.md`

### 2026-03-09 - 视频学习任务完整性检查（新规则！）

**错误**：视频转录任务完成后，未检查音频时长与转录文本时长是否一致，导致只转录了56%的内容就以为任务完成。

**违反的规则**：
- ❌ 未验证任务完整性
- ❌ 未对比音频时长 vs 转录文本时长
- ❌ 未检查转录文本最后时间戳是否匹配音频总时长

**正确做法（视频学习工作流程补充）**：
```markdown
视频学习任务完成后必须执行以下验证：

1. 【时长验证】
   - 获取音频/视频总时长: ffprobe -i audio.mp3 -show_entries format=duration
   - 获取转录文本最后时间戳: grep -oE '\[[0-9]+\.[0-9]+s\]' transcript.txt | tail -1
   - 对比两者是否一致（误差应在10秒内）

2. 【完整性验证】
   - 检查转录文本行数是否合理
   - 检查时间戳是否连续无跳跃
   - 抽样检查文本内容是否完整

3. 【验证通过标准】
   - 转录时长 ≥ 音频时长 × 95%
   - 最后时间戳与音频时长误差 ≤ 10秒
   - 文本内容无大量空白或异常中断
```

**执行时机**：每次视频学习转录完成后，生成PDF之前必须验证！

### 🎬 B站自动生成字幕（2026-03-07 新增）

**重要提示：** B站视频默认可能没有字幕，但UP主或B站系统可以**自动生成字幕**！

#### 如何开启/获取

**方式1：UP主开启**
- UP主在投稿时可选择"自动生成字幕"
- 开启后所有用户都能看到字幕按钮

**方式2：B站AI字幕**
- 部分视频B站会自动生成AI字幕
- 在播放器右下角"字幕"按钮查看

#### 提取方法（开启后）

一旦视频有自动生成字幕，用 yt-dlp 即可提取：

```bash
# 查看可用字幕
yt-dlp "https://b23.tv/xxx" --list-subs

# 提取自动字幕（通常是 ai-generated 或 auto 类型）
yt-dlp "https://b23.tv/xxx" --write-subs --sub-langs "zh-CN,ai-zh" --skip-download
```

#### 为什么重要

| 对比项 | 自动生成字幕 | faster-whisper转录 |
|:---|:---|:---|
| **速度** | 秒级提取 | 分钟级处理 |
| **准确率** | 高（B站官方AI） | 中等 |
| **成本** | 免费 | 免费但耗CPU |
| **适用** | B站有字幕的视频 | 无字幕的视频 |

**结论：B站视频务必先检查是否有自动生成字幕！** 可以省掉大量转录时间。

**3. 音频转录（faster-whisper）**
```python
from faster_whisper import WhisperModel
import os

# 国内镜像（解决网络问题）
os.environ['HF_ENDPOINT'] = 'https://hf-mirror.com'

# 加载模型
model = WhisperModel('small', device='cpu', compute_type='int8')

# 转录
segments, info = model.transcribe("audio.mp3", language="zh", beam_size=5)

# 输出
for segment in segments:
    print(f"[{segment.start:.2f}s] {segment.text}")
```

## 🔄 Gateway重启规范（2026-03-11新增）

| 场景 | 处理方式 | 汇报要求 |

### 2. 紧急恢复标准流程

```markdown
【发现错误】如："Unexpected event order..."
    ↓
【判断】是否为运行时错误？是否影响服务？
    ↓
【是】→ 执行重启：openclaw gateway restart
    ↓
【执行后】→ 立即向组长汇报：
    "组长，网关遇到[错误类型]，我已重启恢复。
     当前状态：[正常/观察中]"
    ↓
【观察】→ 确认服务恢复正常
```

### 3. 禁止行为

❌ **绝不可以：**
- 配置修改后不备份、不汇报就重启
- 紧急重启后**不立即汇报**（让组长自己发现）
- 频繁无理由重启（每天超过1次需说明原因）

✅ **必须做到：**
- 紧急重启后**立即汇报**（5分钟内）
- 说明错误类型和重启原因
- 汇报当前状态（已恢复/仍需观察）

### 4. 常见错误类型

| 错误 | 类型 | 处理方式 |
|:---|:---|:---|
| "Unexpected event order..." | 运行时错误 | 紧急重启 + 立即汇报 |
| "Connection refused" | 服务异常 | 紧急重启 + 立即汇报 |
| 配置解析失败 | 配置错误 | 检查配置，汇报后再重启 |

## 🛠️ 运维经验教训

### 端口冲突处理原则（2026-03-03）
**教训来源**: Star-Office-UI部署过程
**核心原则**: 遇到端口冲突时，应检测可用端口并切换，而不是强制停止现有进程

**正确做法**:
```bash
# 1. 检测端口是否被占用
lsof -i:端口号

# 2. 如果被占用，寻找可用端口
for port in 18791 18792 18793 18794; do
  if ! lsof -i:$port > /dev/null 2>&1; then
    echo "可用端口: $port"
    break
  fi
done

# 3. 使用新端口启动服务
# 修改配置文件或启动参数
```

**错误做法**（避免）:
```bash
# ❌ 不要直接杀掉现有进程
kill -9 $(lsof -ti:端口号)
```

**原因**:
- 现有进程可能是用户其他重要服务
- 强制终止可能导致数据丢失
- 优先选择和平共存而非强制抢占

**现象**: 消息路由混乱或收不到消息

```json

```

**现象**: `ENOTDIR: not a directory` 错误，Agent无响应
**根因**: `.openclaw`被错误创建为**文件**而非**目录**
```bash
# ❌ 错误
-rw-rw-r-- 1 gary gary 0  .openclaw  # 0字节文件

# ✅ 正确
drwxrwxr-x 2 gary gary 4096  .openclaw/  # 目录
```

### 正确配置检查清单
```markdown

3. [ ] workspace/.openclaw/ 是目录（不是文件）
4. [ ] Gateway已重启加载新配置
```
