二、核心要点

2.1 K8s基础概念

历史发展: - 起源于谷歌，2014年开源 - 由CNCF（云原生计算基金会）主导维护 - 已成为云原生生态的核心基础设施

架构组成: - 控制面板（Master节点）: 负责集群入口、请求处理、验证接收 - Node节点: 运行实际工作负载（POD在此运行）

核心资源对象: - POD: K8s最基本的部署单元 - Service: 提供网络访问抽象 - Deployment: 管理POD的部署和更新

核心组件: - API Server: 集群入口，处理所有请求 - ETCD: 分布式键值存储，保存集群状态 - 调度器（Scheduler）: 根据资源情况选择节点部署POD

2.2 CRD与Operator

CRD（自定义资源定义）: - K8s的扩展机制，允许用户通过预定义规范扩展自己的实现 - 通过YAML声明创建、更新、删除自定义对象 - 实现对特定应用的抽象，增强K8s原生能力

Operator: - 由定义CRD的人自行开发 - 负责维护自定义资源的状态和逻辑 - 实现自我修复、自动扩缩容等高级功能

最佳实践: - 明确资源定义，确保意图清晰 - 命名采用反向域名格式（避免冲突） - 状态和期望分离，便于管理

2.3 常见问题排查

问题现象 可能原因 排查方法
Pending状态 资源不足、调度限制、节点亲和性规则不符 kubectl describe pod <name> 查看事件
CrashLoopBackOff 应用配置错误、启动脚本问题、健康检查配置错误 查看日志 kubectl logs <pod>
镜像拉取失败 镜像名称错误、仓库认证失败、网络不可达 检查镜像名、配置代理
容器运行错误 挂载卷不存在、权限不足、启动命令执行异常 查看POD详情和事件

排查工具命令:

# 查看POD状态
kubectl get pods

# 查看POD详情（含事件）
kubectl describe pod <pod-name>

# 查看日志
kubectl logs <pod-name>

# 进入容器
kubectl exec -it <pod-name> -- /bin/bash

Lens 图形化界面，支持多集群管理 日常开发，可视化操作
k9s 终端UI，快捷键操作 喜欢命令行的用户

2.5 Kubeflow简介

定位: - 开源的机器学习平台 - 简化机器学习工作流的部署与管理 - 实现机器学习全流程的可复现、可扩展、自动化

核心组件: - Kubeflow Pipelines: 工作流编排 - Training Operator: 分布式训练（支持PyTorch、TensorFlow、MPI等） - Katib: 超参数调优 - KServe: 模型推理服务

架构位置: - 位于K8s之上 - 位于各种训练框架之下（PyTorch、TensorFlow、MPI等） - 承上启下的作用

分布式训练: - 通过CRD扩展K8s功能 - Training Operator作为底层控制器 - 监听自定义资源状态，驱动训练任务完成 - 负责容错管理、日志收集、资源清理

三、详细内容

3.1 K8s架构详解

控制面板组件: 1. API Server: 所有操作的入口，负责认证、授权、请求处理 2. ETCD: 存储集群所有状态数据（配置、资源状态等） 3. Scheduler: 根据资源需求和策略选择合适节点 4. Controller Manager: 维护集群状态，确保实际状态与期望一致

Node节点组件: 1. Kubelet: 节点代理，管理POD生命周期 2. Container Runtime: 容器运行时（containerd、CRI-O等） 3. Kube-proxy: 网络代理，实现Service的网络转发

3.2 CRD定义示例

Kubeflow PyTorchJob CRD结构:

apiVersion: kubeflow.org/v1
kind: PyTorchJob
metadata:
  name: pytorch-job
spec:
  pytorchReplicaSpecs:
    Master:
      replicas: 1
      template:
        spec:
          containers:
          - name: pytorch
            image: pytorch/pytorch:latest
    Worker:
      replicas: 2
      template:
        spec:
          containers:
          - name: pytorch
            image: pytorch/pytorch:latest

关键字段: - Master: 主节点，负责协调 - Worker: 工作节点，执行训练 - replicas: 副本数 - template: POD模板定义

3.3 问题排查流程

标准排查步骤: 1. 确定问题POD: kubectl get pods 查看状态 2. 查看详细信息: kubectl describe pod <name> 查看Events 3. 查看日志: kubectl logs <pod-name> 4. 进入容器: kubectl exec -it <pod-name> -- /bin/bash 5. 根据错误信息解决: - 资源不足 → 调整资源配置或清理资源 - 镜像问题 → 检查镜像名、认证、网络 - 配置错误 → 修正YAML配置

四、实践建议

4.1 日常使用建议

  1. 优先使用Lens: 图形化界面适合日常开发和问题排查
  2. 掌握kubectl: 脚本自动化和高级操作必备
  3. 善用describe: 遇到问题先describe看Events
  4. 规范命名: 使用有意义的名称，便于管理

4.2 CRD开发建议

  1. 明确资源意图: 定义清晰，避免过度复杂
  2. 版本管理: 使用版本号管理CRD演进
  3. 状态设计: 区分期望状态（Spec）和实际状态（Status）
  4. Operator逻辑: 确保幂等性，处理各种边界情况

4.3 分布式训练建议

  1. 使用Kubeflow: 避免重复造轮子
  2. 合理配置资源: 根据训练需求配置CPU/GPU/内存
  3. 监控和日志: 配置监控和日志收集，便于问题排查
  4. 清理策略: 设置合适的清理策略，避免资源浪费
