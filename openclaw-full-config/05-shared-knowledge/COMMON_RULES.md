## 🎯 核心原则

### 1. Gateway重启安全机制（最高优先级！）

**⚠️ 任何涉及Gateway配置修改和重启的操作，必须严格遵守以下流程：**

#### 四步安全流程

```
步骤1: 【备份】
        ├─ 备份路径: ~/.openclaw/backups/YYYY-MM-DD/
        ├─ 备份命令: 
        │   mkdir -p ~/.openclaw/backups/$(date +%Y-%m-%d)
        │   cp ~/.openclaw/openclaw.json ~/.openclaw/backups/$(date +%Y-%m-%d)/openclaw-$(date +%H%M%S).json
        └─ 确认备份文件存在且大小正常
                    ↓
步骤2: 【确认】
        ├─ 向组长发送消息请示
        │   "组长，我需要修改XXX配置并重启Gateway，是否同意？"
        └─ 等待明确回复"同意"后方可执行
                    ↓
步骤3: 【执行】（组长同意后）
        ├─ 执行配置修改
        ├─ 重启Gateway: gateway restart
        └─ 观察启动日志，确认启动成功
                    ↓
步骤4: 【验证与汇报】
        ├─ 启动成功 → 汇报: "Gateway重启成功，配置已生效"
        └─ 启动失败 → 立即执行回滚
```

#### 启动失败时的回滚流程

```
1. 立即告知组长：
   "Gateway启动失败！错误信息：XXX"

2. 恢复备份：
   cp ~/.openclaw/backups/YYYY-MM-DD/openclaw-XXXXXX.json ~/.openclaw/openclaw.json

3. 重新启动Gateway

4. 汇报回滚结果：
   "已回滚到备份配置，Gateway恢复正常"
```

#### 绝对禁止的行为

- ❌ **未经组长同意擅自重启Gateway**
- ❌ **修改配置后不备份直接重启**
- ❌ **添加未经验证的配置键**
- ❌ **启动失败后不回滚、不汇报**
- ❌ **在高峰时段执行重启操作**

#### 违规后果

违反此规则将导致：
- 整个8-Agent团队服务中断
- 配置丢失，需要手动恢复
- 助手无法响应用户消息
- **严重违规将被记录并追责**

### 3. 定时任务规范（重要！）

**所有定时执行任务必须使用系统后台Cron机制！**

#### 正确做法 ✅

**使用 `openclaw cron add` 命令创建后台定时任务：**

```bash
# 创建定时任务示例
openclaw cron add "0 9 * * *" --agent news --task "推送今日新闻"

# 参数说明
# "0 9 * * *" - Cron表达式（每天9点）
# --agent - 指定执行任务的Agent
# --task - 任务描述
```

**适用场景：**
- ✅ 每日新闻推送
- ✅ 定时提醒/通知
- ✅ 定期数据更新
- ✅ 定时报表生成
- ✅ 任何需要按时执行的任务

#### 禁止做法 ❌

**绝对不允许：**
- ❌ 依赖人工记忆，手动触发
- ❌ 等待用户发消息才执行
- ❌ 使用 sleep 或延时等待
- ❌ 依赖外部定时器或闹钟

#### 为什么必须用Cron？

| 方式 | 问题 | Cron优势 |
|------|------|----------|
| 人工记忆 | 容易遗忘，不可靠 | 系统自动执行 |
| 等待触发 | 可能错过时间点 | 准时触发 |
| 外部工具 | 脱离OpenClaw管理 | 统一管理，可监控 |

#### 如何创建定时任务

**步骤1: 确认任务需求**
```
- 执行时间（如：每天9:00）
- 执行Agent（如：news）
- 任务内容（如：推送新闻）
```

**步骤2: 转换为Cron表达式**
```
每天9:00 → "0 9 * * *"
每周一9:00 → "0 9 * * 1"
每月1日9:00 → "0 9 1 * *"
```

**步骤3: 创建任务**
```bash
openclaw cron add "0 9 * * *" --agent news --task "推送今日新闻"
```

**步骤4: 验证任务**
```bash
openclaw cron list  # 查看所有定时任务
```

#### ✅ 成功配置示例（经过验证）

**场景：定时发送消息给组长**

```json
{
  "agentId": "boss",              // 执行任务的Agent
  "sessionTarget": "isolated",    // 隔离会话执行
  "payload": {
    "kind": "agentTurn",
    "message": "请立即使用message工具给组长(user:ou_xxx)发送消息：内容",
    "timeoutSeconds": 60
  },
  "delivery": {
    "mode": "announce",
    "channel": "feishu",
    "to": "user:ou_xxx"            // 目标用户（该Agent看到的组长ID）
  }
}
```

**关键要点：**
1. ✅ 使用 `agentTurn` payload
2. ✅ 使用 `sessionTarget: "isolated"`
3. ✅ 配置 `delivery.channel: "feishu"`
4. ✅ 配置 `delivery.to: "user:ou_xxx"`（注意：是该Agent看到的组长ID）
5. ✅ **在 message 中明确写**："请立即使用message工具给组长发送消息..."

**工作原理：**
- Cron触发 → 唤醒Agent
- Agent读取message内容
- Agent自己执行message工具发送消息

### 4. 消息发送规范

主动给组长发消息时的**标准流程**：

```
1. 检查是否有活跃会话
   ├─ 有 → 从 inbound_meta.sender_id 读取
   └─ 无 → 从历史会话记录中查找组长的 ID
               ↓
2. 使用获取到的 ID 发送消息
               ↓
3. 消息成功送达
```

**关键原则:**
- ✅ 从**自己**的记录中找到组长的ID
- ✅ 当前无会话时，从历史记录中查找
- ✅ 每个助手使用**不同**的ID（飞书多机器人机制）

### 4. 跨助手协作规范

**当需要其他助手协助时:**
1. 通过 `sessions_send` 发送明确任务指令
2. 指定输出格式和交付位置
3. 跟踪进度，主动汇报给组长

**当收到其他助手求助时:**
1. 立即答复对方的问题
2. 同时**主动汇报**给组长（见规则1）
3. 汇报内容包括：求助者、问题、处理方案、当前状态
