# 消息发送规范

## 主动发消息给组长的标准流程

**生效时间**: 2026-03-06
**验证状态**: ✅ 测试通过

### ID 查找优先级

当需要给组长发消息时，按以下顺序查找 ID：

```
1. 检查是否有活跃会话
   ├─ 有 → 从 inbound_meta.sender_id 读取
   └─ 无 → 从历史会话记录中查找组长的 ID
               ↓
2. 使用获取到的 ID 发送消息
               ↓
3. 消息成功送达
```

### 详细步骤

**方式1：当前有活跃会话**
```javascript
// 从当前会话读取
sender_id = inbound_meta.sender_id  // 格式：ou_xxx
```

**方式2：无活跃会话（从历史记录查找）**
```javascript
// 从会话历史中提取最近的 sender_id
// 路径：~/.openclaw/agents/office/sessions/*.jsonl
// 查找包含组长消息的记录，提取 sender_id
```

### 核心原则

- ✅ **优先**从当前会话读取 `sender_id`
- ✅ **无会话时**，从历史记录中查找
- ✅ 每个助手看到的组长 ID **不同**（飞书多机器人机制）
- ✅ 助手之间互相提醒时也遵循此机制

### 禁止行为

- ❌ 从其他助手/总管的消息中复制 ID
- ❌ 使用固定的/硬编码的 ID
- ❌ 假设所有助手的 ID 都一样
- ❌ 不查找历史记录就放弃发送

### 为什么？

飞书为每个机器人应用分配不同的 `open_id`：
- 方案搭子看到的组长 ID ≠ 办公搭子看到的组长 ID
- 必须使用**自己会话中**的 ID 才能成功发送消息

### 参考

搭子总管维护的映射表仅用于核对，不作为发送依据。
